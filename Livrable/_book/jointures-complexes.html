Chapter 8 Jointures complexes

Problématique

Fusionner deux tables données sur un critère d’appariemment qui soit plus complexe qu’une seule égalité de champs pros sur chacune des tables. Un cas particulier est l’absence de tout critère : chaque ligne de la première table est associée à chacune des lignes de la seconde. Des exemples d’utilisations plus complexes pourraient être la présence d’un mot contenu dans une colonne de la seconde table dans une phrase contenue dans une colonne de la première ou le recouvrement d’intervalles spécifiés par deux colonnes dans chacune des deux tables.

Solution retenue

Faire un produit cartésien entre les deux tables (en clair, si les deux tables ont des tailles respectives N et M, on crée une table de taile NxM), puis sélectionner les croisements à conserver en appliquant la condition. Le produit cartésien fait appel à une fonction spécifique crossing du package tidyr. Pour réduire l’ampleur du produit cartésien, lorsque la condition est un mixte d’égalités et d’autres sous-conditions, il faut procéder en deux étapes : faire la jointure sur les égalités comme au 4.2, puis traités les autres sous-conditions.


Le résultat est une jointure simple (‘inner join’). En cas de besoin une jointure partielle (‘left join’) peut être obtenue par jointure partielle sur clé du résultat avec la table voulue.

Exemples

Le jeu d’essai testé est généré par le programme suivant. Il consiste en une table contenant des points ‘X’, et une table contenant des intervalles ‘Y’ de bornes ‘debut’ et ‘fin’. Le problème constiste à associer à chaque point les intervalles qui le contiennent : debut<=X et X<=fin. On veut conserver les points sans écho parmi les intervalles (jointure partielle).

library(dplyr)
N = 100
M = 10
points <- data.frame(X=round(runif(min=1,max=365,n=N)))
intervalles <- data.frame(Y=1:M,debut=round(runif(min=1,max=365,n=M)),longueur=round(runif(min=1,max=7,n=M)))
intervalles$fin <- pmin(intervalles$debut+intervalles$longueur,365)
resultat <- points %>% 
  tidyr::crossing(intervalles) %>% 
  dplyr::filter(X>=debut & X<=fin) %>% 
  dplyr::right_join(points,"X")

Solutions viables non retenues

Le package sqldf

Le package sqldf permet d’utiliser des requêtes en langage SQL. Malgré la simplicité de la solution, celle ci a été écartée en raison de performances moindres (32 secondes au lieu de 7 secondes sur l’exemple) provenant de la constitution préalable en mémoire d’une base de données SQLite. Cette étape initiale peut d’ailleurs être un obstacle en cas de saturation de la mémoire, mais dans ce cas d’autres solutions que celle préconisée sont de toutes manières nécessaires.

library(sqldf)
resultat <- sqldf("select * from points left join intervalles on X>=debut & X<=fin")

Le package data.table

Le package data.table offre des perfomances supérieures aux autres solutions (sur l’exemple: 5 secondes contre 7 avec la solution préconisée), mais il a été écarté à la fois pour des raisons :
- d’incompatibilités avec le reste des outils préconisés : les performances du package tiennent en particulier au fait qu’il ne respecte pas le fonctionnement habituel des fonctions sous R et qu’il peut modifier les objets qu’on lui confie,
- d’une syntaxe spécifique faussement proche de celle du R de base,
- d’une moins grande généralité : les conditions peuvent inclure plusieurs sous conditions, mais celles ci sont liées par des ET et ne peuvent que porter sur deux variables (une de chaque table) avec des opérateurs limités aux opérateurs de comparaison de type supérieur/inférieur.

library(data.table)
a <- points
b <- intervalles
name1 <- setdiff(names(setDT(b)), names(setDT(a))) 
resultat <- a[b, (name1) := mget(name1), on = .(X >= debut, X <= fin)]

Le package fuzzyjoin

L’utilisation du package fuzzyjoin, conçu pour ce type de cas est écartée en raison de sa complexité de mise en oeuvre et de performances nettement inférieures (43 secondes sur l’exemple). En outre la documentation du package pour des cas non triviaux (comme le cas pris en exemple) est très nettement insuffisante et l’analyse du code a rèvélé que l’exécution reposait sur un produit cartésien (mais limité aux variables utilisées dans la condition).

library(fuzzyjoin)
resultat <- fuzzy_join(points,intervalles
           , multi_by = list(x="X",y=c("debut","fin"))
           , multi_match_fun = function(x,y) (x>=y[,"debut"])&(x<=y[,"fin"])
               , mode = "left")

R de base

La solution basée sur du R de base est écartée à la fois en raison de sa complexité et de performances nettement inférieures (45 secondes sur l’exemple). Néanmoins elle pourrait avoir un sens dans un contexte de grandes tables où le passage par un produit cartésien serait réhibitoire.

En R de base, le principe constiste à boucler sur l’ensemble des valeurs des points, et pour chacune de celles ci à produire une table des intervalles répondant à la question, puis à fusionner toutes les tables résultat. Cette approche présente plusieurs difficultés qui peuvent être contournées mais au détriment de la lisibilité du programme et de ses performances.
Dans la condition, on ne peut utiliser qu’une seule variable de la table ‘points’ : if faudrait utiliser mapply au lieu de lapply en citant toutes les variables nécessaires de la table ‘points’, ce qui rend une généralisation difficile.
Le résultat est une jointure simple (‘inner join’) d’où vont être écartés les points sans écho dans la table de intervalles : il faudrait utiliser replyr::replyr_bind_rows au lieu de l’appel à ‘rbind’ via ‘do.call’. Les performances sont alors sensiblement dégradées (110 secondes sur l’exemple).
Il n’est pas possible de se passer de la syntaxe de base à l’intérieur de la condition (par exemple avec ‘within’) sans très sensiblement dégrader les performances (157 secondes sur l’exemple).

resultat <- do.call("rbind",
    lapply(points$X,
    FUN=function(x)
      { c <- intervalles[(x>=intervalles$debut & x<=intervalles$fin),]
        if (nrow(c)>0) data.frame(x,c) else NULL
      }
    ))
